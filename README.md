# Recall ‚Äî an adaptive study assistant

Paste your notes or name a topic. Recall turns it into flashcards or a quiz,
generated by an LLM (Groq) as structured JSON ‚Äî not a chat transcript you
have to scroll through.

The thing that makes this more than a flashcard-printer: cards aren't
"correct/wrong once and done." Each card tracks a **mastery streak** ‚Äî rate
it "Got it" twice in a row and it graduates out of the deck. "Shaky" or
"Missed" sends it back into rotation, and you get a short message telling
you exactly what happened before the next card appears. Progress isn't
all-or-nothing either ‚Äî the progress bar moves on every correct answer, not
just when a card fully masters, so early progress actually feels like
progress.

## Setup

```bash
npm install
cp .env.example .env      # then paste your Groq API key into .env
npm start                  # runs at http://localhost:5173
```
Get a free Groq API key at https://console.groq.com/keys ‚Äî no cost for the
model used here (`llama-3.3-70b-versatile`).

The dev server *is* the backend: `vite.config.js` runs a small middleware
(`devApiPlugin`) that intercepts `POST /api/generate` inside the same Node
process and calls Groq server-side. The API key is read from `.env` on the
server and never reaches the browser bundle. `api/generate.js` is the same
logic packaged as a Vercel serverless function for a real deploy ‚Äî both
entry points share `lib/groq.js` and `lib/schema.js`, so there's one source
of truth for the AI call and validation instead of two copies that could
drift apart.

## Handling bad AI output

| Failure mode | Where it's caught | What the user sees |
|---|---|---|
| Non-JSON / malformed response | `validateStudySet` in `lib/schema.js`, `JSON.parse` wrapped in try/catch | One automatic repair retry with a stricter prompt; if that also fails, a clear error + retry button |
| Right JSON, wrong shape (missing fields, wrong types) | Zod schema validation (`StudySetSchema`) | Same repair-retry path, then error state |
| Empty item list | Explicit length check in `validateStudySet` | "No study items could be generated ‚Äî try adding more detail" |
| Slow / hanging request | Client-side 25s timeout via `AbortController` in `useStudySet.js` | Loading indicator, then a timeout-specific error message |
| Newer request started before an older one resolves | `requestIdRef` counter in `useStudySet.js` ‚Äî any response for a stale id is silently dropped | The screen only ever reflects the most recent request; no flicker back to old data |
| Network / provider failure | try/catch around `fetch` in `lib/groq.js` and `api.js` | Error state with retry, no crash |
| Missing API key | Checked explicitly in `generateStudySet` before calling Groq | Clear setup-error message instead of a generic failure |

## What I used AI for (Claude)

I want to be specific here rather than write one vague sentence, because
the specifics are the actual evidence of what I understand vs. what I
leaned on help for.

- **Scaffolding**: Claude generated the initial Vite + Tailwind config
  boilerplate (`tailwind.config.js`, `postcss.config.js`, the Google Fonts
  setup in `index.html`) ‚Äî mechanical setup, not core logic.
- **Backend plumbing**: I hadn't written a Vite dev-server middleware
  before. Claude drafted the first version of `devApiPlugin` in
  `vite.config.js`, and I had it explain the middleware API
  (`server.middlewares.use`, reading the raw request stream) line by line
  before I used it ‚Äî I wanted to actually understand how a request reaches
  that function before shipping it.
- **Prompt design**: the Groq system prompt in `lib/groq.js` was AI-drafted
  and then iterated on manually after testing what actually broke in
  practice ‚Äî the model initially wrapped JSON in markdown code fences on
  some runs, which is why `response_format: { type: "json_object" }` and an
  explicit "no code fences" instruction are both in the final prompt.
- **Debugging the race condition**: I knew *that* a fast second request
  could get overwritten by a slow first response landing late, but I hadn't
  built the fix before. Claude walked me through the `requestIdRef` counter
  pattern in `useStudySet.js` ‚Äî I re-implemented and tested it myself
  rather than pasting it in as-is.
- **UI direction**: I gave Claude a reference screenshot of a product
  landing page I liked (mesh gradient background, thin serif display type,
  pill-shaped buttons) and asked for the same visual language applied to
  this app. The color tokens, gradient values, and component styling in
  `tailwind.config.js` / `src/index.css` came out of that back-and-forth.
- **UX iteration on the mastery loop**: my first version of the mastery
  logic only moved the progress bar once a card was *fully* mastered, and
  gave no feedback on a wrong answer ‚Äî it just silently moved to the next
  card. I found this confusing once I actually used it, described the
  problem, and worked through the fix (granular progress based on streak
  points, and a feedback message with a short delay before advancing) with
  Claude's help, then implemented it myself.

What I designed and built without relying on AI to write the first draft:
the schema shape in `lib/schema.js`, the retry-on-validation-failure control
flow in `lib/groq.js`, the mastery-streak state machine in `StudyDeck.jsx`
(including the requeue-to-back-of-queue behavior), and the failure-mode
table above. Those are the parts I can walk through and extend live without
notes, because I made the actual decisions in them, not just the styling.

## Known limitations

- Quiz mode auto-rates correctness against `correctIndex`; flashcard mode is
  self-rated (honor system) ‚Äî there's no way to verify a "Got it" claim on
  a flashcard beyond trusting the user.
- No persistence yet ‚Äî refreshing the page loses progress on the current
  deck.
- The repair-retry in `lib/groq.js` only retries once, deliberately ‚Äî a
  model that consistently fails validation surfaces a clear error instead
  of retrying indefinitely against the API.
- Tested primarily against `llama-3.3-70b-versatile` on Groq. Smaller or
  local models (e.g. via Ollama) will hit the repair-retry path more often
  since they're less consistent at structured output ‚Äî the failure-handling
  table above is what absorbs that difference.
